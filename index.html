Register

  • First, the researcher will need to create an account on the WiNEST platform and an initial project which serves as a sort of folder for profiles, experiments, users, and other associated resources.
  • It is expected that the user will enter his/her institutional email address so that we can easily verify the person.
  • We may be able to simplify the onboarding process for existing CloudLab and GENI users.

Registered

  • After the account is created, it must be manually reviewed by WiNEST staff. A private admin page will streamline and simplify the review process.
  • An email will be sent out to the researcher once the account is approved or to explain the reason for rejection.

Login

  • Once the researcher account is approved, he/she may use the login page.

Dashboard

  • The dashboard page shows various platform resources created by or accessible to the researcher. Here, we show panels for active experiments and experiment profiles.
  • For a new account, nothing has been created yet, but helpful links direct the researcher to launch one of the publicly available experiment profiles or to create a new experiment profile.

Dynamic Backhaul Profile

  • The profile creation page asks for basic descriptive information as well as a way to supply, generate, and visualize the Rspec.
  • Here we show an experiment that establishes wireless links between specified towers using our dynamic backhaul capability as well as computing resources at two points in the network.
  • The experiment execution engine will enforce certain feasibility constraints (# links <= # radios) and resolve conflicts with existing experiments.

Dynamic Backhaul Experiment

  • The experiment status page will show detailed information about resources allocated to the experiment.
  • For each compute node, we provide information for SSH access, a web-based console for quick access, systems logs, and a reboot mechanism in case problems occur.
  • Additionally, since this experiment allocated custom wireless links, we provide detailed link status information.

Sensor Profile

  • This experiment profile pulls in data streams from some sensors and uses edge computing to preprocess the data.
  • The network topology shows two kinds of computing resources: one is a VM provisioned in the NOC "cloud", and the other is a container provisioned in close proximity to the sensors.

Sensor Experiment

  • The experiment status page shows detailed information about resources allocated to the experiment.
  • For each compute node, we provide information for SSH access. Since node-2 is a container rather than a full VM, it needs to be handled slightly differently.
  • Additionally, since this experiment requested access to sensor data streams, extra information about the sensors is shown on this page such as the sensor type, update interval, and last reading.

Sleep Scheduling Profile

  • This experiment profile reserves an Open Cellular base station and two connected client units (CPEs).
  • The clients have some edge computing capabilities through Paradrop, and this experiment launches a container on one of them for collecting measurements.

Sleep Scheduling Experiment

  • The experiment status page shows detailed information about resources allocated to the experiment.
  • For each compute node, we provide information for SSH access. Since node-2 is a container rather than a full VM, it needs to be handled slightly differently by passing through a different port.
  • Since this experiment creates wireless links between a base station and clients, detailed information about those links is available.

Documentation

Forum

  • This page shows a forum for asking questions as well as notifying the community of important changes.
  • We will use an open source solution such as Discourse.
  • Forums are a great tool because they encourage users to answer each other's questions and preserve information in searchable form to lessen burden on the platform staff.
